# Vibemeet — техническая спецификация

> Документ описывает целевую систему, архитектурные решения, требования и
> компромиссы. Язык реализации — Go 1.21. Документ актуален на момент
> начального релиза.

## 1. Назначение и контекст

Vibemeet — self-hosted система видеоконференций для команд и сообществ,
которые не готовы или не могут отдавать видеопоток сторонним SaaS-сервисам
(требования регуляторов, приватность, контроль инфраструктуры).

Целевой пользователь — техническая команда, способная развернуть стек на
собственном сервере или в собственном облаке, использующая `docker compose`
как примитив развёртывания.

Минимальный сценарий: пользователь открывает ссылку в браузере, попадает в
комнату, видит и слышит других участников, может включить демонстрацию
экрана и обмениваться сообщениями в чате. Регистрация не требуется.

## 2. Глоссарий

| Термин | Значение |
| --- | --- |
| Комната (room) | Логическая единица созвона: множество участников + общая медиа-сессия LiveKit + чат. |
| Анонимная комната | Комната без владельца, создаётся одним HTTP-запросом, идентичность участника — cookie. |
| Авторизованная комната | Комната с владельцем (`host_user_id`), приглашениями, ролями, постоянным чатом. |
| SFU | Selective Forwarding Unit — медиа-сервер, который ретранслирует RTP-потоки между участниками. В системе роль SFU выполняет LiveKit. |
| ICE candidate | Сетевой эндпоинт, который WebRTC-клиент предлагает партнёру для установления p2p-соединения. |
| Access / refresh token | Пара JWT: короткий access (минуты), долгий refresh (дни). Refresh хранится хешированным в БД и может быть отозван. |

## 3. Цели и нецели

### Цели

- Полностью self-hostable стек: один `docker compose up` поднимает всю систему.
- Анонимный сценарий — без БД-сайдэффектов на пользователя, без email, без подтверждений.
- Авторизованный сценарий — JWT-аутентификация с возможностью отзыва токенов.
- Браузер + LAN-устройства как первоклассные клиенты (mobile-friendly UI).
- Прозрачная конфигурация через переменные окружения; обязательные секреты
  отказывают в старте, а не запускаются с дефолтами.
- Структурное логирование (`log/slog`) и health-эндпоинты, пригодные для
  внешних балансировщиков и систем мониторинга.

### Нецели

- Транскрипция, summarization, AI-фичи.
- Мобильные нативные клиенты (iOS/Android).
- Запись медиа на сервере. LiveKit умеет, но дефолтный compose-стек не
  поднимает egress-узел — это осознанное решение, см. §11.
- Биллинг, тарификация, multi-tenant изоляция на уровне инстанса.
- Federation между разными инстансами Vibemeet.
- E2E-шифрование медиа (LiveKit маршрутизирует SRTP, ключ известен серверу).

## 4. Архитектура

### 4.1. Топология

```
                  ┌──────────────────────────────┐
                  │           Браузер            │
                  │  (HTML + JS + LiveKit SDK)   │
                  └───────┬──────────────┬───────┘
                          │ HTTPS        │ WebRTC (UDP/TCP)
                          ▼              ▼
                  ┌──────────────┐   ┌──────────────┐
                  │    Nginx     │   │   LiveKit    │
                  │ (reverse     │   │    (SFU)     │
                  │  proxy, TLS) │   │              │
                  └──────┬───────┘   └──────┬───────┘
                         │ HTTP             │ gRPC (server-side)
                         ▼                  │
                  ┌──────────────┐          │
                  │   Go API     │◀─────────┘
                  │   (Gin)      │
                  └───┬──────┬───┘
                      │      │
              SQL ────┘      └──── Redis (кэш, rate-limit, anon-chat)
                      ▼
                ┌──────────┐
                │ Postgres │
                └──────────┘
```

### 4.2. Компоненты

| Компонент | Назначение | Обоснование выбора |
| --- | --- | --- |
| Go API (Gin) | REST API, бизнес-логика, выпуск LiveKit-токенов, аутентификация. | Gin — минимальный router без runtime-магии; `pgx` даёт типобезопасный доступ к Postgres без ORM-сюрпризов. |
| Nginx | Терминация TLS, статика, обратный прокси к API. | Стандарт де-факто; конфиг короткий, поведение предсказуемое. |
| LiveKit | SFU: маршрутизация RTP, ICE, TURN, грантовая модель доступа. | Опенсорс, лицензия Apache 2.0, активный мейнтейнер, продакшен-grade в десятках компаний. |
| Postgres 15 | Persistence для users, rooms, refresh-tokens, audit log. | ACID, JSONB-поля для `settings`, проверенные миграционные паттерны. |
| Redis 7 | TTL-кэш токенов комнат, чат для анонимных комнат, rate-limit счётчики. | Низкая латентность; естественные TTL для эфемерных данных. |
| Pion WebRTC | Серверный захват экрана как fallback для клиентов без screen capture. | Чистый Go, без CGO-зависимостей на runtime-стороне приложения. |

### 4.3. Потоки данных

**Сигнализация (signalling).**
Браузер → Nginx → Go API. Go API проверяет, что комната существует и
активна, генерирует LiveKit access token с грантами (`RoomJoin`,
`CanPublish`, `CanSubscribe`), возвращает токен и URL LiveKit-сервера.

**Медиа.**
Браузер ↔ LiveKit напрямую (UDP/TCP). API не участвует. LiveKit
ретранслирует RTP между участниками, делает simulcast и selective
forwarding. ICE-кандидаты строятся от `HOST_IP`, передаваемого в LiveKit
через env.

**Чат.**
Авторизованные комнаты: сообщения пишутся в Postgres (`chat_messages`).
Анонимные комнаты: сообщения хранятся в Redis-списке с TTL 6 часов;
ключ — `chat:room:<uuid>:messages`. После TTL сообщения исчезают; это
осознанное поведение для приватности (§9).

## 5. Функциональные требования

### 5.1. Анонимный сценарий (без аккаунта)

| ID | Требование |
| --- | --- |
| F-A1 | Создание комнаты одним POST-запросом без аутентификации. |
| F-A2 | Присоединение к комнате по UUID-ссылке без регистрации. |
| F-A3 | Идентификация участника через secure cookie (UUID participant_id). |
| F-A4 | Выпуск LiveKit-токена по participant_id и room_id. |
| F-A5 | Чат с TTL 6 часов и лимитом 1000 сообщений на комнату. |

### 5.2. Авторизованный сценарий

| ID | Требование |
| --- | --- |
| F-U1 | Регистрация по email + пароль (bcrypt cost 10). |
| F-U2 | Логин выдаёт access (15 мин) и refresh (30 дней). |
| F-U3 | Refresh-эндпоинт ротирует токен; старый refresh инвалидируется. |
| F-U4 | Logout помечает refresh-токен отозванным в БД. |
| F-U5 | Создание комнаты, приглашения, роли (host/co-host/participant). |
| F-U6 | Постоянный чат, сохраняемый в Postgres. |
| F-U7 | Настройки пользователя (язык, дефолтное имя, аватар). |

### 5.3. Системные

| ID | Требование |
| --- | --- |
| F-S1 | `/api/v1/health` — liveness, всегда `200 OK`. |
| F-S2 | `/api/v1/health/ready` — readiness, проверяет Postgres + Redis. |
| F-S3 | `/server-info` — раскрытие `LIVEKIT_FRONTEND_URL` для frontend bootstrapping. |
| F-S4 | Graceful shutdown по `SIGTERM` с таймаутом 15 секунд. |
| F-S5 | Структурное логирование в stdout (JSON) с уровнем из `LOG_LEVEL`. |

## 6. Нефункциональные требования

### 6.1. Производительность

- API p95 latency для read-эндпоинтов: **< 100 ms** на reference-стенде
  (4 vCPU / 8 GB RAM, локальные Postgres и Redis).
- Создание токена LiveKit: **< 20 ms** (HMAC-SHA256, без I/O).
- Один LiveKit-узел при `HOST_IP` с публичным IP: расчётно
  до **50 одновременных участников** в комнате и до **500 одновременных
  участников** в инстансе (зависит от bitrate и режима simulcast).

### 6.2. Масштабируемость

- Go API — stateless, горизонтально масштабируется за L7-балансировщиком.
- LiveKit поддерживает multi-node режим (redis-backed signalling), но
  дефолтный compose-стек разворачивает один узел.
- Postgres и Redis — single-instance в дефолтной конфигурации; для прод
  с высокой нагрузкой нужны managed managed-сервисы (RDS, ElastiCache).

### 6.3. Надёжность

- Все compose-сервисы имеют `restart: unless-stopped`.
- API не запускается, если обязательные env-переменные не заданы
  (`fail-fast`). Это сделано через docker-compose `${VAR:?}` синтаксис
  и через явную валидацию в `internal/config`.
- Healthcheck зависимости Postgres/Redis перед стартом API — компонент
  ждёт готовности БД и кэша через `condition: service_healthy`.

### 6.4. Безопасность

См. §9.

### 6.5. Совместимость

- Браузер: Chrome ≥ 90, Safari ≥ 15, Firefox ≥ 88. WebRTC API,
  `getDisplayMedia`, `getUserMedia`.
- Docker Compose v2 (синтаксис v3.8 в compose-файле).
- Go 1.21+ для пересборки из исходников.

## 7. Модель данных

### 7.1. Основные таблицы

| Таблица | Назначение | Ключевые поля |
| --- | --- | --- |
| `users` | Учётные записи. | `id (uuid)`, `email (unique)`, `password_hash`, `created_at`. |
| `rooms` | Авторизованные комнаты. | `id (uuid)`, `host_user_id`, `livekit_room_name`, `status`, `settings (jsonb)`. |
| `room_participants` | Участники комнат. | `room_id`, `user_id`, `role`, `joined_at`, `left_at`. |
| `room_invites` | Приглашения по ссылке. | `link_token`, `expires_at`, `max_uses`, `uses_count`. |
| `refresh_tokens` | Активные refresh-токены. | `user_id`, `token_hash`, `expires_at`, `revoked_at`. |
| `chat_messages` | Постоянный чат. | `room_id`, `sender_user_id`, `body`, `created_at`. |
| `audit_logs` | Аудит событий. | `event_time`, `actor_user_id`, `event_type`, `payload (jsonb)`. |
| `anonymous_rooms` | Анонимные комнаты. | `id (uuid)`, `livekit_room_name`, `created_at`. |

Анонимный чат не имеет таблицы — хранится только в Redis.

### 7.2. Жизненный цикл

- `rooms.status`: `scheduled → active → ended`. Переход в `active` — на
  первом успешном `join`.
- `refresh_tokens.revoked_at`: при logout или ротации; rows не удаляются,
  чтобы сохранить audit trail.
- `anonymous_rooms`: TTL не запрашивается явно, но фактический lifetime
  ограничен TTL чата в Redis (6 часов с последнего сообщения).
- `chat_messages` (анонимные): TTL 6 часов в Redis, после чего теряются
  невосстановимо.

## 8. Программные интерфейсы

Полный референс — `docs/API.md`. Здесь — границы и контракты.

### 8.1. REST API

Base path: `/api/v1`.

- `POST /auth/{register, login, refresh, logout}` — учётные записи.
- `GET /users/me`, `PATCH /users/me/settings` — пользователь.
- `POST /rooms`, `GET /rooms`, `GET /rooms/:id` — авторизованные комнаты.
- `POST /rooms/:id/invites` — приглашения.
- `POST /rooms/:id/join`, `POST /rooms/:id/leave` — анонимный сценарий.
- `POST /rooms/:id/media/token` — выпуск LiveKit-токена.
- `GET /rooms/:id/chat/messages`, `POST /rooms/:id/chat/messages` — чат.

Формат ошибок:
```json
{ "error": "human-readable message", "code": "OPTIONAL_CODE" }
```

### 8.2. Внутренние границы

- `internal/handler` → `internal/service` (бизнес-логика).
- `internal/service` → `internal/repository` (persistence).
- `internal/repository` → Postgres / Redis. Repositories возвращают
  `domain.*` типы, не `*pgx.Rows`. Это сделано, чтобы вынос на другую БД
  не требовал переписывать сервисный слой.

## 9. Безопасность и приватность

### 9.1. Модель угроз (краткая)

| Угроза | Митigation |
| --- | --- |
| Кража access-токена | TTL 15 минут; HMAC-SHA256 с серверным секретом; HTTPS обязателен. |
| Кража refresh-токена | Хранится хешированным; ротация при использовании; возможен отзыв. |
| Brute-force паролей | Per-IP rate-limit на `/auth/login` (Redis-backed). |
| Перебор UUID комнат | UUIDv4 (122 бита энтропии); rate-limit на endpoints. |
| MITM на сигнализации | TLS терминация в Nginx (`docs/DEPLOYMENT.md`). |
| Утечка чата | Анонимный чат TTL 6 часов; авторизованный — хранится постоянно, но за auth. |
| Утечка ICE-кандидатов | `HOST_IP` явно конфигурируется; нет автоматического раскрытия внутренних IP. |

### 9.2. Секреты

- Все секреты — через env. Compose-файлы используют `${VAR:?}` для
  обязательных переменных: контейнер не стартует без них.
- `env.sample` содержит только dev-дефолты с предупреждением.
- `env.prod.example` — пустые значения; коммит реального `.env.prod`
  блокируется `.gitignore`.

### 9.3. Аудит

Все доменные события (создание комнаты, join, leave, kick, инвайт)
записываются в `audit_logs` с актором и контекстом в `payload (jsonb)`.

### 9.4. Что система НЕ делает

- Не пишет PII в логи (email хешируется при логировании, ID логируются).
- Не делает E2E-шифрования (см. нецели).
- Не хранит медиа.

## 10. Развёртывание и эксплуатация

### 10.1. Окружения

- **Dev**: `docker-compose.yml` + `env.sample`. Порты — стандартные.
  Self-signed TLS не нужен.
- **Prod**: `docker-compose.prod.yml` + `.env.prod`. Порты нестандартные
  (HTTP 18080, LiveKit 17880-17882) — чтобы коэксистировать с другими
  сервисами на хосте.

Подробно — `docs/DEPLOYMENT.md`.

### 10.2. Зависимости от внешней инфраструктуры

- Сетевой стек: для WebRTC нужны открытые UDP-порты (LiveKit: 7882/17882 для RTP).
- TLS-сертификат: получается отдельно (Let's Encrypt) и монтируется в
  `./ssl/`. Renewal — внешний (cron + certbot).

### 10.3. Бэкапы

В дефолтном compose volume для Postgres — `vibemeet_postgres_data`.
Стратегия бэкапа описана в `docs/DEPLOYMENT.md` (логические дампы
`pg_dump` + офсайт-копия).

### 10.4. Мониторинг

- Health-эндпоинты: `/api/v1/health` (liveness), `/api/v1/health/ready`
  (readiness).
- Логи — stdout в JSON; ожидается доставка во внешний агрегатор
  (Loki/Datadog/Splunk) на уровне Docker logging driver.
- Метрики не экспортируются на момент начального релиза — см. §12.

## 11. Ограничения и компромиссы

| Компромисс | Причина | Альтернатива |
| --- | --- | --- |
| Без записи медиа | LiveKit egress требует отдельного узла и AWS S3-совместимого стораджа. Для портфолио-релиза — оверкилл. | Добавить `livekit-egress` сервис в compose. |
| Без E2E-шифрования | LiveKit маршрутизирует SRTP; ключи известны серверу. Включение SFU-режима E2EE требует существенной работы в клиенте. | Использовать `livekit-meet` с включённым E2EE. |
| Один LiveKit-узел | Multi-node режим требует Redis-cluster для signalling и тщательной настройки TURN-relay. | См. LiveKit docs → distributed deployment. |
| Без миграций (есть только `init_db.sql`) | На уровне начального релиза схема стабильна; добавление миграционного фреймворка (`golang-migrate`, `goose`) — следующий шаг. | См. §12. |
| Чат анонимных комнат теряется | Приватность по умолчанию для незарегистрированных. | Авторизованный сценарий хранит чат в Postgres. |

## 12. Дорожная карта

Приоритеты на следующие итерации:

- **Миграции схемы**: переход с `init_db.sql` на `golang-migrate`.
- **Метрики**: Prometheus-эндпоинт `/metrics` (RED для HTTP, gauges для
  активных комнат и участников).
- **OpenTelemetry**: трейсинг от Nginx до Postgres.
- **Frontend testing**: Playwright-сценарии для anonymous flow.
- **E2E-шифрование**: исследование совместимости LiveKit SFU + insertable streams.
- **Запись комнат**: интеграция `livekit-egress` с S3-совместимым стораджем.
- **i18n**: вынос UI-строк фронтенда из HTML в JSON-словари.
- **Permissions UI**: интерфейс для управления приглашениями и ролями.
